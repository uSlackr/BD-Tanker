# Tanker Fermentation Manager - Design Notes

## Goal

Provide a local-first way to package the temperature history for a complete fermentation so a brewer can review one fermentation chart and compare it with other batches of the same or different beers.

The system should support:

- start/end tracking for a fermentation batch
- beer name / batch metadata
- tank association
- charting actual temperature over the fermentation window
- comparison charts across batches
- future export/backup of fermentation packages

## Design Constraints

- All site data must remain local to the Home Assistant box / brewery site.
- No cloud dependency for operational data.
- Encrypted backups are acceptable.
- Home Assistant should remain the main UI/control surface.
- InfluxDB remains the time-series source of truth for temperature history.
- Home Assistant recorder data may be useful, but should not be the primary long-term fermentation package store.

## Recommended Architecture

Build a local **Tanker Fermentation Manager** service deployed as a Home Assistant add-on.

The add-on should run on the HA box and provide:

- local API service
- local metadata database, likely SQLite
- InfluxDB query client
- optional local web UI for charting and comparisons
- endpoints called by Home Assistant scripts/rest_commands

### Why a Home Assistant Add-on

Compared with a loose Docker container, a Supervisor add-on gives us:

- HA-native install/update lifecycle
- local-only deployment model
- easier backup/restore with HA backups
- cleaner configuration/secrets handling
- local service discovery from HA
- productizable deployment path for future Tanker sites

## Data Storage

### SQLite Metadata Database

Use SQLite for fermentation metadata. It is small, local, portable, and easy to include in encrypted HA backups.

Example table:

```sql
fermentations
- id
- site
- tank_name
- beer_name
- batch_name
- start_time
- end_time
- temp_entity_id
- setpoint_entity_id optional
- notes
- created_at
- updated_at
```

InfluxDB should remain the source for raw time-series readings. The fermentation DB stores the time window and metadata needed to query the relevant data.

## Fermentation Package Concept

A fermentation package is not necessarily a copied blob of all temperature samples at first. Initially it can be:

```json
{
  "id": "...",
  "site": "brudaddy",
  "tank": "30F1",
  "beer_name": "Example Pale Ale",
  "batch_name": "2026-05-Example",
  "start_time": "2026-05-05T12:00:00-04:00",
  "end_time": "2026-05-12T12:00:00-04:00",
  "temp_entity_id": "sensor.bd_remote_b_30f1_temp"
}
```

Chart data is generated by querying InfluxDB for the stored entity and time window.

Later, we can add immutable exports containing both metadata and sampled series data:

```json
{
  "fermentation": { "...": "metadata" },
  "series": {
    "actual_temp": [],
    "setpoint": [],
    "cooling_state": []
  }
}
```

## API Sketch

Initial local API endpoints:

```text
POST /api/batches/start
POST /api/batches/end
GET  /api/batches/active
GET  /api/batches/history
GET  /api/batches/{id}
GET  /api/batches/{id}/chart-data
GET  /api/compare?batch_ids=...
```

Example start request from Home Assistant:

```json
{
  "site": "brudaddy",
  "tank": "30F1",
  "beer_name": "Pale Ale",
  "batch_name": "optional batch name",
  "temp_entity_id": "sensor.bd_remote_b_30f1_temp"
}
```

The backend should enforce truth. For example, it should reject starting a second active batch on the same tank unless the previous batch has been ended.

## Home Assistant UI Approach

Home Assistant remains the brewer-facing control surface.

### Phase 1: Add controls to each fermenter card

On each tank/fermenter card in the existing site dashboard:

- Start Batch button
- End Batch button
- Current batch status
- Elapsed fermentation time
- optional View Chart link/button

The existing `input_text.<tank>_beer` helpers can provide the beer name used when starting a batch.

### Phase 2: Fermentation Manager dashboard

Create a dedicated HA dashboard or add-on panel showing:

- active fermentations
- completed fermentations
- batch search/filter
- chart view
- comparison selector
- notes/edit metadata

## Charting / Comparison

For a single fermentation:

- query InfluxDB from `start_time` to `end_time`
- plot actual temperature
- optionally overlay setpoint, cooling call/valve state, ambient, gravity, etc.

For comparisons:

- align series by elapsed fermentation time rather than wall clock
- show X-axis as fermentation age: Day 0, Day 1, Day 2...
- preserve actual timestamp in tooltips
- support comparing the same beer across batches or different beers

## Security / Local-First Posture

- Service should bind only to the local HA/add-on network.
- No telemetry.
- No external ingress required.
- Optional local API token shared with HA.
- SQLite DB and config should be included in encrypted HA backups.
- InfluxDB remains local to the site.

## Suggested Implementation Phases

### Phase 0 - Prototype

- Build minimal local service with SQLite.
- Hardcode one site/tank mapping for BruDaddy.
- Implement start/end/list active/history.
- Query InfluxDB for one fermentation chart.

### Phase 1 - HA Control Surface

- Add HA `rest_command` or scripts for start/end.
- Add buttons to BruDaddy tank cards.
- Add REST sensors for active batch state.

### Phase 2 - Chart UI

- Add local web UI in the add-on or iframe panel.
- Single-batch chart first.
- Add batch comparison after the core flow is solid.

### Phase 3 - Productize

- Convert prototype into a proper HA Supervisor add-on.
- Add configuration for site name, InfluxDB connection, tank/entity mappings.
- Add backup/export support.
- Document install/update process for Tanker sites.

## Open Questions

- Should batch IDs be user-entered, autogenerated, or both?
- What additional signals should be captured besides actual temperature?
  - setpoint
  - glycol valve/cooling state
  - ambient temperature
  - gravity
  - brewer notes/events
- Should completed fermentation packages be immutable after closing?
- How should chart snapshots/exports be represented for long-term archival?
- Should HA expose active fermentation status as REST sensors or via a future custom integration?
